# AeroWatch — Neighbourhood Pollution Hotspot Map

A front-end prototype for **CleanAir & Clear Streets**: a hyperlocal air-quality
system that fuses citizen photo reports, ground sensor readings, and
satellite imagery to surface pollution hotspots city-wide monitors miss,
forecast the next 24 hours, and route municipal crews to exactly where
they're needed.

This is a **static, self-contained demo** — pure HTML/CSS/JS, no build step,
no backend. All sensor readings, forecasts, and alerts are generated by a
simulation engine (`js/data.js`) so the whole flow is explorable without any
real infrastructure behind it. It's meant as a working prototype / pitch
artifact, with clear seams for plugging in real data later (see below).

## Run it

No install needed — just open `index.html` in a browser, or serve the folder:

```bash
python3 -m http.server 8000
# then visit http://localhost:8000
```

Requires an internet connection only for the Google Fonts CDN link in
`css/style.css`; everything else runs offline.

## Pages

| Page             | Purpose                                                                 |
|------------------|--------------------------------------------------------------------------|
| `index.html`     | Landing page: the problem, how the pipeline works, feature overview.     |
| `dashboard.html` | Live hex sensor-mesh map, per-cell AQI gauge, 24h forecast chart, sensor list. |
| `report.html`    | Citizen flow: photo upload, location, category/severity, submit a report. |
| `ops.html`       | Municipal ops console: alert queue, acknowledge/dispatch/resolve, resource roster. |

## How the simulation works

- **`js/data.js`** defines 16 neighbourhood cells on a hex grid, each with a
  land-use type (`traffic`, `industrial`, `dump`, `residential`, `green`), a
  baseline AQI, and a volatility factor. `currentReading()` derives a live
  number per cell (diurnal traffic curve + noise); `forecast()` projects 24h
  forward in 3-hour steps, occasionally injecting a "hidden hotspot" spike for
  dump/industrial cells with no ground sensor — modelling exactly the kind of
  event a city-wide average would miss.
- **Citizen reports** (`report.html` → `AeroData.addReport`) and **alerts**
  (`ops.html` → `AeroData.getAlerts` / `updateAlert`) persist to
  `localStorage` so state carries across pages/reloads on the same device.
  This is a plain static site (not a Claude.ai artifact), so `localStorage`
  is safe to use here.
- **`js/hexmap.js`** renders the sensor mesh as SVG hexagons (axial
  coordinates → pixel space), colour-coded by AQI category, with a pulsing
  ring on cells the model flags as hotspots and a small dot marking cells with
  no physical sensor (photo/satellite-only coverage).
- **`js/gauge.js`** draws an analog-instrument-style AQI dial; **`js/charts.js`**
  draws the 24h forecast line chart with an alert-threshold reference line.

## Wiring in real data later

The simulation layer is intentionally isolated so it can be swapped for a
real backend without touching the UI:

- Replace `AeroData.CELLS` + `currentReading()`/`forecast()` with API calls to
  your sensor ingestion + ML forecasting service (same shape: `{ aqi, peak,
  forecast[] }` per cell).
- Point `AeroData.addReport()` at a real endpoint (photo upload + geotag)
  instead of `localStorage`.
- Feed `AeroData.getAlerts()` from your dispatch/ticketing system, and wire
  `updateAlert()` to real acknowledge/dispatch/resolve API calls.
- Swap the synthetic hex grid for real geography — either keep a hex-binned
  aggregation (good for sensor-density-agnostic display) or switch
  `hexmap.js` to a tile-based map (e.g. Leaflet/Mapbox) using the same
  per-cell data shape.

## Stack

Vanilla HTML / CSS / JavaScript. No frameworks, no build tools, no external
JS dependencies — only a Google Fonts CSS import. Everything renders as
inline SVG generated at runtime, so there are no image assets to manage.
